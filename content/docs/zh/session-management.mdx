## 概述

Chat2API 支持两种对话模式以适应不同使用场景。会话管理在多次 API 调用之间维护对话上下文。

<Card title="快速参考" icon={<Zap />}>
  在 **高级设置** → **代理** → **会话管理** 中配置
</Card>

## 会话模式

### 单轮模式

每次请求都是独立的，不保留上下文。

<Callout type="info">
  默认模式。适用于无状态 API 调用。
</Callout>

**特点：**

| 特性 | 描述 |
|------|------|
| 上下文 | 请求之间不保留 |
| 会话 ID | 返回空字符串 |
| 资源占用 | 最小 |
| Provider 会话 | 可选在聊天结束后删除 |

**适用于：**

- 简单问答
- 翻译任务
- 文本生成
- 一次性查询

**超时后删除：**

在单轮模式下，启用 `deleteAfterTimeout` 可在聊天结束后自动删除 Provider 端的对话。这有助于：

- 节省 Provider 存储
- 保护隐私
- 清理未使用的对话

### 多轮模式

在同一会话中保持对话上下文。

**特点：**

| 特性 | 描述 |
|------|------|
| 上下文 | 请求之间保留 |
| 会话 ID | 每个会话唯一 |
| 资源占用 | 较高（存储历史） |
| Provider 会话 | 保持以维持连续性 |

**适用于：**

- 对话式 AI 应用
- 多步骤任务
- 依赖上下文的查询
- 客服机器人

## 配置

在 **高级设置** → **代理** → **会话管理** 中配置：

| 选项 | 描述 | 默认值 | 范围 |
|------|------|--------|------|
| 模式 | 单轮或多轮 | 单轮 | - |
| 会话超时 | 会话过期前的分钟数 | 30 | 1-1440 |
| 每会话最大消息数 | 保留的最大消息数 | 100 | 1-500 |
| 每账户最大会话数 | 最大并发会话数 | 10 | 1-10 |
| 超时后删除 | 聊天后删除 Provider 会话（仅单轮） | 禁用 | - |

## 使用会话

### 创建会话

在请求中包含 `session_id` 时会自动创建会话：

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your-api-key" \
  -d '{
    "model": "DeepSeek-V3.2",
    "messages": [
      {"role": "user", "content": "我叫小明"}
    ],
    "session_id": "user-123-session"
  }'
```

### 继续会话

使用相同的 `session_id` 继续对话：

```bash
curl http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer your-api-key" \
  -d '{
    "model": "DeepSeek-V3.2",
    "messages": [
      {"role": "user", "content": "我叫什么名字？"}
    ],
    "session_id": "user-123-session"
  }'
```

模型会记住上下文并回复："你叫小明。"

### 会话 ID 格式

| 要求 | 描述 |
|------|------|
| 唯一性 | 每个用户/对话使用唯一 ID |
| 格式 | 推荐使用 UUID 或用户特定标识符 |
| 大小写 | 会话 ID 区分大小写 |

## 会话生命周期

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Created   │ ──▶ │   Active    │ ──▶ │   Expired   │ ──▶ │   Cleaned   │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

| 阶段 | 说明 |
|------|------|
| Created (创建) | 首次带 session_id 的请求 |
| Active (活跃) | 每次请求保持活跃（可配置超时） |
| Expired (过期) | 达到超时时间 |
| Cleaned (清理) | 自动删除 |

<Callout type="warning">
  **重要提示**：如果您在官方 Web 页面上清除了会话历史记录，请务必在 Chat2API 中手动清理对应的 Session，否则可能导致对话上下文不一致。
</Callout>

### 状态

| 状态 | 描述 |
|------|------|
| `active` | 会话正在使用，上下文保留 |
| `expired` | 达到超时，标记为待清理 |
| `cleaned` | 会话已从内存中移除 |

## 会话限制

### 消息数限制

当会话达到 `maxMessagesPerSession` 时：

1. 最旧的消息首先被移除
2. 保留最近的上下文
3. 会话继续正常工作

### 会话数限制

当账户达到 `maxSessionsPerAccount` 时：

1. 最旧的非活跃会话被移除
2. 可以创建新会话
3. 活跃会话不受影响

## 最佳实践

### 会话 ID 管理

1. **使用 UUID**：为每个对话生成唯一 ID
2. **用户关联**：将会话 ID 与用户标识符绑定
3. **一致格式**：使用一致的命名规范

### 配置调优

| 使用场景 | 超时时间 | 最大消息数 | 最大会话数 |
|----------|----------|------------|------------|
| 快速问答 | 5-10 分钟 | 20-50 | 5 |
| 客服支持 | 30-60 分钟 | 100-200 | 10 |
| 长对话 | 120+ 分钟 | 200-500 | 3 |

### 资源管理

1. **监控使用**：定期检查会话列表
2. **设置合理限制**：平衡功能和资源
3. **主动清理**：需要时使用清空功能

## 故障排除

### 会话不工作

<Callout type="warning">
  验证会话模式设置为 **多轮** 以保留上下文。
</Callout>

**检查清单：**

- [ ] 会话模式为多轮
- [ ] 请求中包含 `session_id`
- [ ] 会话未过期
- [ ] 多次请求使用相同的 `session_id`

### 上下文丢失

**可能原因：**

| 原因 | 解决方案 |
|------|----------|
| 会话过期 | 增加超时时间或重新创建会话 |
| 达到消息限制 | 增加最大消息数设置 |
| 会话被删除 | 检查是否被手动删除 |

### 会话过多

**解决方案：**

1. 减少 `maxSessionsPerAccount`
2. 减少 `sessionTimeout` 以加快清理
3. 使用界面中的清空功能
4. 在应用中实现会话清理逻辑

### Provider 会话未删除（单轮模式）

如果启用了 `deleteAfterTimeout` 但 Provider 会话仍然存在：

- 检查 Provider 是否支持会话删除
- 验证账户是否有适当权限
- 查看日志中的删除错误
